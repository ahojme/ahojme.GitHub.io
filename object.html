Issue Management Plan

A workflow module to propose, plan, and track remediation • 2023

Client

An intergovernmental organisation in the security sector

Product

Web application (internal tool)

My role

Lead UX/UI designer

Year

2023

Goal

Design the layout and interactions of a module allowing to propose, plan, and monitor issues remediation (a part of a broader initiative to digitalise complex business workflows)

Users

Organisation employees


Challenges

  • Displaying data-dense content on a single page
  • Managing interdependencies between the main object (remediation plan proposal) and sub-objects (schedules, implementation plans, requests, messages)
  • Designing for multiple user groups with varying access rights and duties
  • Research limitations due to restricted access to some users and data

Tools and methods

Focus groups, interviews, user flows, user story map, Object-Oriented UX, concept testing, unmoderated testing

1. Discovery


To gather insights on requirements, user needs, and pain points, I conducted focus groups and individual interviews. Due to the restricted nature of the data, methods like field studies or analytics were not feasible.

Key Findings:

  • Users are responsible for analysing data to diagnose problems, propose solutions, and request their implementation.
  • They monitor process progress and manage associated risks, and they need rapid risk detection and response. Time pressure and handling large volumes of data are significant challenges.
  • The core object (proposal) follows a complex path involving multiple approvals and collaboration across user groups. Moreover, sub-objects have their own set of dependencies and deadlines.



2. Problem definition

I created an object map using the Object-Oriented UX approach to identify dependencies between the main object and sub-objects, along with a user story map. These helped me define key issues and frame the design challenges. To encourage creativity, I formulated them in a "How Might We" (HMW) format


How might we help users:

  • stay informed about the process's progress and status
  • manage risks of incidents and delays
  • gain clarity on upcoming tasks and the impact of their actions
  • document modifications of the proposals and plans
  • collaborate with internal and external stakeholders
  • ensure restricted information is accessible only to authorised users
  • reduce mistakes and accelerate the business processes

3. Process of solution development

Based on insights gained in the previous phase, the initial assumption was that various data sets are interdependent. To help users make informed decisions, sub-objects must be accessible and viewed simultaneously. I created initial wireframes and high-level user flows to explore various data presentation options.



Next, I developed more detailed user flows, which helped define key features for objects and sub-objects, including:

  • Proposal content: Text areas, attribute display, inline forms, schedule tables
  • Proposal status: Status badges, tags, and tasks list with deadlines
  • Collaboration: Comments, requests, and approval workflows
  • Tracking changes: history and version comparison

They prompted further iterations of the solution. During this phase, I worked closely with the developers' team and frequently contacted the stakeholders.

4. Final design

I used a card-based layout. The main proposal and its attachment (Schedule) are placed in the left column. To make the table easier to read, the right pane can be collapsed.

Tasks to be performed are located at the top left, so they are always visible. Communication-related features are placed in tabs.

Forms are placed inline so that the user filling out a form can see data in other parts of the interface.

5. Next steps and learnings

  • We will implement notifications to help users respond to important incidents on time.
  • Testing revealed that a specific user group, which rarely uses the application, struggles to locate a particular form placed in a tab. In the next iteration, I focused on improving the feature visibility for this user group and paid attention to similar issues.
  • Testing also showed that some users did not understand why certain actions were unavailable, often blaming their lack of business knowledge. This highlighted a need to address the first usability heuristic: Visibility of System Status. In the next iteration, I explored ways to support users in understanding the system's intricacies, helping them feel confident and in control.
